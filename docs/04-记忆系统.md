# 记忆系统

## 两层记忆架构

SonettoHere 的记忆系统分为两层：

```
┌─────────────────────────────────────────────┐
│  短期记忆 (InMemoryChatMessageHistory)         │
│  - 当前会话的消息历史                          │
│  - LangChain 内置，不做 token 裁剪             │
│  - 1M 上下文窗口，消息随会话累积               │
├─────────────────────────────────────────────┤
│  记忆叙事 (Memory Narrative)                   │
│  - 跨会话的自然语言记忆                         │
│  - 每轮对话后 LLM 增量更新                      │
│  - 存储介质：config/personas/MEMORY.md         │
└─────────────────────────────────────────────┘
```

---

## 第一层：短期记忆（InMemoryChatMessageHistory）

直接使用 LangChain 内置的 `InMemoryChatMessageHistory`（`langchain_core.chat_history`），不再做自定义包装。

### 核心数据结构

```python
from langchain_core.chat_history import InMemoryChatMessageHistory

memory = InMemoryChatMessageHistory()  # 本质是 list[BaseMessage]
```

提供 `add_message`、`add_messages`、`clear` 方法和 `messages` 属性。选用的 LLM 上下文窗口高达 1M tokens，因此不需要 token 裁剪——消息在会话内自然累积，会话结束时清理。

---

## 第二层：记忆叙事（Memory Narrative）

在 [memory/narrative.py](../memory/narrative.py) 中。

### 工作原理

每轮对话结束后，将本轮**裸对话消息**（不做任何预处理）连同旧的 `MEMORY.md` 一起送给 LLM，让 LLM 在现有记忆基础上融合新信息，写回 `config/personas/MEMORY.md`。

```
回合结束:
  turn_messages (裸消息)
      │
      └── update_narrative(llm, turn_messages)
            │
            ├── 冷启动 (MEMORY.md 不存在):
            │     turn_messages → LLM → 生成 MEMORY.md
            │
            └── 常态 (MEMORY.md 已存在):
                  turn_messages + 旧 MEMORY.md → LLM → 更新 MEMORY.md
```

### 为什么用裸消息而不是提取的摘要？

旧系统先用 Extractor 把对话压缩成 summary，再拿 summary 去检索或生成。这导致信息损失叠加，且两个模块耦合。叙事模块直接读裸消息，让 LLM 看到原始对话的全部细节。

### 叙事 LLM Prompt

LLM 通过四个 CRUD 工具（`create_memory` / `read_memories` / `update_memory` / `delete_memory`）管理记忆条目，而非直接撰写文本。

**冷启动**（MEMORY.md 不存在时）：LLM 从空状态开始，用 `create_memory` 逐条添加事实，自行选择合适的分区名。

**常态更新**（MEMORY.md 已存在时）：LLM 先调用 `read_memories` 查看现有条目（每条带唯一 ID 和分区），对比新对话后增/改/删条目。系统提示词会列出建议分区名作为参考，但 LLM 在现有分区不合适时可自行新建（1-4字中文名词）。

**分区建议**（系统提示词中提供）：
- "身份" — 基本身份信息，极少变化
- "音乐" — 虚拟歌手、声库、歌曲、专辑、创作者
- "品味" — 电影、美食、UP主、品牌偏好等
- "地点与路径" — 具体地点和文件系统路径
- "瞬间" — 即时观察和感受，短暂轮替
- "时效待办" — 有截止日期的事项，到期后应删除

### 叙事注入系统提示词

在 [agent/prompts.py](../agent/prompts.py) 中，`build_system_prompt()` 将 `MEMORY.md` 的内容注入到 `"## 我对用户的记忆"` 段落。`build_enhanced_prompt()` 每轮重新调用 `build_system_prompt()`，确保叙事始终是最新的。

### 叙事内容示例（分区格式）

```markdown
# 长期记忆索引
- [身份](#身份)
- [音乐](#音乐)
- [瞬间](#瞬间)

---

## 身份
- 用户就读于北京邮电大学沙河校区，从事网络空间安全领域工作。
- 用户是广东人，家乡在江门市蓬江区。

## 音乐
- 用户特别喜欢洛天依。
- 用户最喜欢的歌曲是洛天依的《海边城》。

## 瞬间
- 用户正在吃汉堡王的外卖（芝士牛堡）。
```

### 文件存储

| 文件 | 说明 |
|------|------|
| `config/personas/MEMORY.md` | AI 自动生成的叙事，每轮更新，gitignore |
| `config/personas/USER.md` | 用户手动填写的自述，gitignore |
| `config/personas/USER.example.md` | 用户自述模板，git 跟踪 |
| `memory/user_init.py` | 首次运行从 example 复制 USER.md |

## 记忆系统的协同工作

一次完整对话回合中，两层记忆的协作流程：

```
用户输入 "帮我查一下明天天气"
        │
        ▼
1. 组装系统提示词
   build_enhanced_prompt()
   → 读取 USER.md (用户自述)
   → 读取 MEMORY.md (记忆叙事)
   → 拼接 AGENTS.md + SOUL.md + USER.md + MEMORY.md
        │
        ▼
2. Agent 推理 → 执行工具 → 回答
        │
        ▼
3. 回合结束 → update_narrative()
   裸消息 + 旧 MEMORY.md → LLM → 新 MEMORY.md
```

这样，Agent 在本次对话中使用了过去的记忆（通过叙事注入），同时本次对话的关键信息又被 LLM 融合进叙事，供下次使用——形成一个自我改进的闭环。
