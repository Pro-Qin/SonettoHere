# 记忆系统

## 三层记忆架构

SonettoHere 的记忆系统分为三层，分别解决不同的记忆需求：

```
┌─────────────────────────────────────────────┐
│  短期记忆 (ShortTermMemory)                   │
│  - 当前会话的消息历史                          │
│  - 基于 tiktoken 的自动裁剪                    │
│  - 阈值：28,000 tokens                        │
├─────────────────────────────────────────────┤
│  长期记忆 (LongTermMemory)                     │
│  - 跨会话的关键信息存储                         │
│  - 关键词 + 置信度 + 时间衰减评分检索            │
│  - 存储介质：JSON 文件                         │
├─────────────────────────────────────────────┤
│  偏好画像 (Preference)                         │
│  - 从长期记忆中聚合用户的稳定偏好                │
│  - 存储介质：独立的 JSON 画像文件                │
└─────────────────────────────────────────────┘
```

---

## 第一层：短期记忆（ShortTermMemory）

在 [memory/short_term.py](../memory/short_term.py) 中。

### 核心数据结构

```python
class ShortTermMemory:
    def __init__(self, max_tokens: int = 28000):
        self._store = InMemoryChatMessageHistory()   # LangChain 内置
        self._max_tokens = max_tokens
        self._encoding = tiktoken.encoding_for_model("gpt-4")
```

`InMemoryChatMessageHistory` 是 LangChain 提供的内存级消息历史存储。它本质上是一个 `list[BaseMessage]`，提供 `add_message`、`add_messages`、`clear` 方法。

### Token 自动裁剪机制

```python
def add_message(self, message: BaseMessage) -> None:
    self._store.add_message(message)
    self._trim()          # ← 每次添加消息后自动检查

def _trim(self) -> None:
    msgs = self._store.messages
    while self._count_tokens(msgs) > self._max_tokens and len(msgs) > 2:
        msgs.pop(0)       # ← 从头部删除最旧的消息

def _count_tokens(self, messages: list[BaseMessage]) -> int:
    total = 0
    for m in messages:
        content = m.content if isinstance(m.content, str) else str(m.content)
        total += len(self._encoding.encode(content))
        total += 4         # ← 每条消息的元数据开销估算
    return total
```

### 裁剪策略的要点

1. **FIFO 淘汰**：从消息列表头部（最旧的消息）开始删除，保留最近的对话上下文
2. **保留最少 2 条**：即使超限也保留至少 2 条消息，避免上下文完全丢失
3. **token 估算**：使用 `tiktoken` 的 `gpt-4` 编码器（与 DeepSeek 的 tokenizer 高度相似），每条消息额外加 4 个 token 估算元数据开销
4. **阈值选择**：28,000 tokens 是一个权衡值——DeepSeek Chat 的上下文窗口为 128K，28K 的阈值在保留足够上下文的同时避免了消息列表无限膨胀导致的延迟和成本问题

### 为什么不用 LangChain 内置的 Memory 方案？

LangChain 提供了 `ConversationBufferWindowMemory`（保留最近 K 轮）、`ConversationSummaryMemory`（用 LLM 压缩历史）等方案。本项目选择自建的原因：

- **Window Memory**："K 轮对话"是粗粒度的衡量，不同轮次的 token 量差异巨大（一轮可能只有 50 tokens，另一轮可能 5000 tokens）
- **Summary Memory**：引入额外的 LLM 调用来做摘要，增加延迟和成本，且可能丢失精确的参数信息
- **Token 级别的精确控制**：基于 tiktoken 实时计算，确保在任何时刻都不超出限制

## 第二层：长期记忆（LongTermMemory）

在 [memory/long_term.py](../memory/long_term.py) 中。

### 数据来源

长期记忆的数据来自 `extracted/extracted_history.json`，这是一个 JSON 数组，每条记录由对话回合的**提取器**生成：

```json
[
  {
    "summary": "用户查询了明天的天气，Agent 成功返回了北京昌平区的天气预报",
    "keywords": ["天气", "北京", "昌平区", "明天"],
    "correction": "",
    "mistake": "Agent 初次使用了模糊地址搜索查天气，实际应该直接用天气工具",
    "user_preferences": [
      {"key": "默认城市", "value": "北京", "confidence": 0.9, "evidence": "用户多次查询北京天气"}
    ],
    "error_patterns": [
      {"type": "tool_selection", "tool": "fuzzy_address_search", "trigger": "查天气时", "detail": "错误地用模糊搜索代替天气查询"}
    ],
    "anti_patterns": [
      {"type": "tool_selection", "tool": "get_current_weather", "instruction": "天气查询直接使用 get_current_weather"}
    ],
    "confidence": 0.85,
    "timestamp": "2026-05-09 14:30:00",
    "session_id": "20260509_143000_abc12345",
    "source": "cli"
  }
]
```

### 三维度评分检索

```python
def retrieve_long_term_context(query: str, top_k: int = 10):
    for record in history:
        kw_score = _keyword_score(query, record)       # 关键词匹配 (权重 0.4)
        conf_w = _confidence_weight(record)            # 置信度加权 (权重 0.2)
        time_w = _time_decay(record.get("timestamp"))  # 时间衰减   (权重 0.4)
        total = kw_score * 0.4 + conf_w * 0.2 + time_w * 0.4
```

#### 维度 1：关键词命中评分 (0.4)

```python
def _keyword_score(query: str, record: dict) -> float:
    keywords = record.get("keywords", [])
    query_lower = query.lower()
    hits = sum(1 for kw in keywords if kw.lower() in query_lower or query_lower in kw.lower())
    return min(hits / max(len(keywords), 1), 1.0)
```

双向匹配：query 包含 keyword，或 keyword 包含 query。这覆盖了"用户说'天气'"匹配到 keyword `["北京天气"]`，以及"用户说'北京昌平天气'"匹配到 keyword `["天气"]` 两种场景。

#### 维度 2：置信度加权 (0.2)

直接使用提取器输出的 `confidence` 字段（0.0 ~ 1.0）。LLM 在提取时对自己不确定的记录给出低置信度，避免低质量记忆污染检索结果。

#### 维度 3：时间衰减 (0.4)

```python
def _time_decay(timestamp_str: str) -> float:
    elapsed_hours = (datetime.now() - record_time).total_seconds() / 3600
    return 0.5 ** (elapsed_hours / 168)  # 半衰期 7 天
```

指数衰减，半衰期 7 天。一条一周前的记录得分减半，两周前的记录得分降至 1/4。权重 0.4 意味着时间因素对检索结果有显著影响——Agent 更倾向于参考近期的记忆。

### 检索结果的分类输出

```python
return {
    "error_rules": [...],       # 最多 10 条 — 需要规避的错误
    "preference_rules": [...]   # 最多 12 条 — 用户的偏好/习惯
}
```

纠偏规则（`error_rules`）和偏好规则（`preference_rules`）被分别注入系统提示词的不同段落，让 LLM 在推理时能区分"不该做什么"和"用户喜欢什么"。

## 第三层：偏好画像（Preference）

在 [memory/preference.py](../memory/preference.py) 中。

### 从长期记忆中聚合稳定偏好

```python
def sync_preferences(history: list[dict]) -> None:
    profile: dict[str, dict] = {}
    for item in history:
        for pref in item.get("user_preferences", []):
            key = str(pref.get("key", "")).strip()
            confidence = float(pref.get("confidence", 0.5))
            prev = profile.get(key)
            if prev is None or confidence >= float(prev.get("confidence", 0.0)):
                profile[key] = {                           # 保留置信度最高的那条
                    "value": pref.get("value", ""),
                    "confidence": confidence,
                    "updated_at": item.get("timestamp", ""),
                }
    # 写入 user_preference_profile.json
```

聚合策略：对于同一个偏好 key（如"默认城市"），保留历史上置信度最高的那条记录。这避免了早期低质量提取覆盖后期更准确的结果。

### 在 CLI 中使用

```python
def _build_enhanced_prompt(self, user_input: str) -> str:
    prompt = self.system_prompt
    stable = get_stable_preferences()
    if stable:
        lines = [f"- {k} = {v.get('value', '')}" for k, v in stable.items()]
        prompt += f"\n\n## 稳定偏好\n" + "\n".join(lines[:5])
    return prompt
```

偏好画像在每次对话回合前被注入系统提示词，让 Agent"记住"用户的基本信息（如默认城市、称呼、空闲时间等），无需用户反复说明。

## 提取器（Extractor）：记忆的来源

在 [memory/extractor.py](../memory/extractor.py) 中。

### 工作原理

每个对话回合结束后，CLI 调用 `extract_from_messages` 将当前回合的 messages 送入 LLM，要求 LLM 以结构化 JSON 格式提取其中的关键信息。

```python
EXTRACTION_PROMPT = """你是一个专业的文本分析师，能够根据Agent对话记录提取出关键信息。
你的输出格式为json，键名如下：
{
    "summary": "提取后的摘要",
    "keywords": ["关键词1", "关键词2"],
    "correction": "用户修正Agent错误的信息",
    "mistake": "Agent出现的错误信息",
    "user_preferences": [{"key":"称呼","value":"Miso","confidence":0.9}],
    "user_habits": [{"habit":"倾向简短回答","confidence":0.7}],
    "error_patterns": [{...}],
    "anti_patterns": [{...}],
    "confidence": 0.0-1.0
}"""
```

### 容错设计

```python
def extract_from_messages(messages, llm, max_retries=2):
    for _ in range(max_retries + 1):
        try:
            raw = llm.generate(prompt)
            parsed = json.loads(raw)
            return _normalize(parsed)
        except (json.JSONDecodeError, Exception):
            continue

    # 降级：手动拼接摘要
    fallback = " | ".join(
        f"{m.get('role', '?')}: {str(m.get('content', ''))[:80]}"
        for m in messages[-6:]
    )
    return _normalize({"summary": fallback, "keywords": [], "confidence": 0.2})
```

- 最多重试 2 次（共 3 次尝试）
- 全部失败则降级为手动拼接最近 6 条消息的前 80 个字符
- 置信度设为 0.2，标记为低质量提取

### 存储上限管理

```python
history.append(record)
history = history[-300:]  # 保留最近 300 条
```

300 条上限配合时间衰减（半衰期 7 天，30 天后的记录得分降至 ~5%），确保长期记忆文件不会无限增长，同时旧记录自然老化。

## 记忆系统的协同工作

一次完整对话回合中，三层记忆的协作流程：

```
用户输入 "帮我查一下明天天气"
        │
        ▼
1. 长期记忆检索
   retrieve_long_term_context("明天天气")
   → 命中 keyword "天气"、"北京"、"昌平区"
   → 返回 {"error_rules": ["天气查询直接用 weather 工具"], "preference_rules": [{"key":"默认城市","value":"北京昌平区"}]}
        │
        ▼
2. 偏好画像注入
   get_stable_preferences()
   → {"称呼": "Miso", "默认地址": "北京昌平区"}
        │
        ▼
3. 增强系统提示词 → Agent 推理 → 执行 → 回答
        │
        ▼
4. 回合结束 → 提取器提取摘要 → 写入 extracted_history.json
        │
        ▼
5. 偏好同步（可选，定期执行）
   sync_preferences(history) → 更新 user_preference_profile.json
```

这样，Agent 在本次对话中使用了过去的记忆，同时本次对话的关键信息又被写入记忆，供下次使用——形成一个自我改进的闭环。
